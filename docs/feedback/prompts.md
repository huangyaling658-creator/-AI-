# 反馈分析模块 — Prompt 设计文档

本文档记录反馈分析模块使用的所有 AI Prompt，便于后续调优和维护。

源代码位置：`src/lib/feedback/prompts.ts`

---

## Prompt 1：维度推荐

**触发时机：** 用户确认数据预览后，AI 读取前 20 条样本，2秒内返回推荐维度
**调用参数：** maxTokens=2048, temperature=0.3

```
你是一位资深用户研究专家。以下是用户上传的反馈数据样本（共{N}条）。

请分析这些数据的特征，推荐 3-6 个最有价值的分析维度。

对每个维度说明：
1. name: 维度名称（简短，如"情感倾向"、"问题分类"）
2. description: 一句话说明分析这个维度的价值
3. supported: true/false，当前数据是否支持此维度
4. reason: 为什么支持或不支持

常见维度供参考（不限于此，根据数据特征灵活推荐）：
- 情感倾向：正面/负面/中立分布
- 问题分类：功能缺陷、体验问题、性能问题、需求建议等
- 优先级评估：按频次和情感强度综合评分
- 功能模块归属：反馈涉及产品的哪个模块/功能
- 用户画像标签：如果数据含用户信息
- 时间趋势：如果数据含时间信息

严格以 JSON 数组格式返回，不要包含其他文字：
[{"name":"...","description":"...","supported":true,"reason":"..."}]

## 反馈样本数据
1. {反馈1}
2. {反馈2}
...
```

**设计要点：**
- 提供参考维度但不限死，鼓励AI根据数据灵活推荐
- `supported` 字段让前端可以把不支持的维度标灰
- `reason` 字段帮助用户理解为什么某维度不适用

## Prompt 2：分析执行（核心 Prompt）

**触发时机：** 用户选择维度后点击"开始分析"
**调用参数：** maxTokens=8192, temperature=0.2
**分批策略：** 超过 500 条时自动分批调用后合并

```
你是一位资深用户研究专家，请对以下用户反馈数据进行深度分析。

## 数据清洗规则（分析前先执行）
1. 去除完全重复的反馈
2. 过滤无意义内容（纯表情、单字回复如"好""嗯""OK"、无关广告等）
3. 合并表述不同但含义相同的反馈（保留一条，计入频次）

## 分析维度
1. {用户选择的维度1}
2. {用户选择的维度2}
...

## 输出要求
严格以 JSON 格式返回，不要包含其他文字。JSON 结构如下：

{
  "summary": {
    "total": 原始总条数,
    "valid": 清洗后有效条数,
    "deduplicated": 去重后条数
  },
  "categories": [
    { "name": "分类名", "count": 数量, "percent": 百分比(保留1位小数), "feedbackIds": [0,3,7] }
  ],
  "sentiments": {
    "positive": { "count": 数量, "percent": 百分比 },
    "negative": { "count": 数量, "percent": 百分比 },
    "neutral": { "count": 数量, "percent": 百分比 }
  },
  "topIssues": [
    { "issue": "问题描述", "count": 出现次数, "sentiment": "negative/positive/neutral", "keywords": ["关键词1","关键词2"] }
  ],
  "keywords": [
    { "word": "关键词", "count": 出现次数 }
  ],
  "feedbacks": [
    { "id": 原始序号(从0开始), "content": "原始内容", "category": "所属分类", "sentiment": "positive/negative/neutral", "sentimentScore": -1到1的浮点数, "keywords": ["关键词"] }
  ]
}

注意：
- topIssues 最多返回 10 条，按 count 降序
- keywords 最多返回 20 个，按 count 降序
- feedbacks 中的 id 对应原始数据的序号
- percent 都保留 1 位小数
- 所有分类名称使用中文

## 反馈数据（共{N}条）
[0] {反馈内容0}
[1] {反馈内容1}
...
```

**设计要点：**
- 数据清洗在分析前执行，去除噪音数据
- `feedbackIds` 支持前端"原文溯源"功能（点击分类展开原始反馈）
- `sentimentScore` 是 -1 到 1 的浮点数，比简单三分类更精细
- 每条反馈都有完整标注，可直接导出为带标注的 CSV/Excel
- temperature 设为 0.2，追求稳定性和一致性

## Prompt 3：多文件字段识别

**触发时机：** 用户上传多份表格文件，系统需要识别哪列是反馈内容
**调用参数：** maxTokens=1024, temperature=0.2

```
你是数据分析专家。以下是用户上传的多份反馈数据文件，每份文件有不同的列名。

请识别每份文件中哪一列最可能是"用户反馈内容"（如评论、意见、建议等）。

严格以 JSON 数组格式返回，不要包含其他文字：
[{"fileName":"文件名","feedbackColumn":"列名"}]

## 文件信息
### {文件名1}
列名: 用户ID, 评论内容, 评分, 日期
示例数据:
001 | 加载太慢了 | 2 | 2024-01-15
002 | 很好用 | 5 | 2024-01-16

### {文件名2}
列名: ticket_id, description, status
示例数据:
T001 | 登录一直失败 | open
T002 | 搜索不到结果 | closed
```

**设计要点：**
- 只展示前 3 行样本，足够AI判断但不浪费 token
- 不同渠道的列名差异很大（中文/英文/缩写），AI 语义理解比关键词匹配更准

## Prompt 4：图片 OCR

**触发时机：** 用户上传图片文件（截图、照片等）
**调用方式：** 多模态消息（text + image_url），利用模型视觉能力

```
请识别这张图片中的所有文字内容。如果图片包含用户反馈、评论或意见类的内容，请逐条提取。

以 JSON 数组格式返回每条反馈文本：
["反馈1", "反馈2", ...]

如果图片中没有可识别的反馈内容，返回空数组 []
```

**设计要点：**
- 简短明确，避免视觉模型产生过多幻觉
- 返回 JSON 数组，便于与其他文本反馈统一合并

---

## 调优建议

1. **分类不准确时：** 在分析 Prompt 的"分析维度"部分增加具体的分类定义和示例
2. **情感判断偏差时：** 可增加 few-shot 示例（给出几条反馈及其正确的情感标注）
3. **关键词太泛时：** 在 Prompt 中增加"关键词应具体到产品功能或用户痛点，避免过于笼统"
4. **数据量大时：** 当前分批阈值为 500 条，可根据实际 token 限制调整
